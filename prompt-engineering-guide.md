## 一、万能提示词公式（记住这 3 个）

### 公式 1：3-Part Rule（命中率提升 2.4x）

```
上下文（Context） + 动作（Action） + 成功标准（Success Criterion）
```

| 元素 | 弱 ❌ | 强 ✅ |
|------|-------|-------|
| 上下文 | "在这个代码里" | "在 `src/api/users.ts` 第 34 行" |
| 动作 | "改进一下" | "抽取为纯函数，保持接口不变" |
| 标准 | （缺失） | "现有测试必须全部通过" |

### 公式 2：CO-STAR（复杂任务，内容生成）

```
C - Context（背景）：项目背景、技术栈、业务场景
O - Objective（目标）：要达成什么具体结果
S - Style（风格）：技术文档/教程/报告/代码
T - Tone（语气）：专业严谨/通俗易懂
A - Audience（受众）：初级开发者/架构师/PM
R - Response（输出格式）：Markdown/JSON/代码+注释
```

### 公式 3：RISEN（多步骤任务，代码审查/流程）

```
R - Role（角色）：你是一个资深后端架构师
I - Instructions（指令）：审查这段代码的安全性和性能
S - Steps（步骤）：1) 先理解业务逻辑 2) 检查注入点 3) 分析复杂度 4) 给出修改建议
E - End goal（最终目标）：输出一份包含风险等级和修复方案的安全审查报告
N - Narrowing（约束）：只关注 SQL 注入和 XSS，忽略样式问题
```

## 二、日常高频提示词模板

### 代码生成
```
## 上下文
- 项目：[项目名]，技术栈：[React 18 / TypeScript / Prisma]
- 文件：@[文件路径]
- 相关代码：@[依赖文件1] @[依赖文件2]

## 任务
实现 [具体功能描述]，要求：
1. 遵循项目现有的 [命名规范/目录结构/错误处理模式]
2. 输入：[类型]，输出：[类型]
3. 边界情况：[列出 2-3 个]

## 约束
- 不引入新的依赖
- 保持与现有代码风格一致
- 添加必要的类型定义

## 输出
- 修改的文件列表
- 每个文件的关键改动说明
```

### Bug 修复
```
## 问题
- 现象：[具体描述]
- 复现步骤：[步骤 1→2→3]
- 环境：[Node 20 / Chrome 120 / etc.]

## 分析要求
1. 先阅读 @[相关文件] 理解逻辑
2. 推断根因（不要猜，基于代码分析）
3. 提出修复方案（至少 2 个备选）
4. 推荐最佳方案并说明理由

## 约束
- 最小改动原则
- 修复后输出验证方法
```

### 代码审查
```
你是 [资深前端架构师]。
审查 @[文件路径]：
1. 安全性：检查 [XSS / CSRF / 注入 / 敏感信息泄露]
2. 性能：检查 [不必要的重渲染 / 内存泄漏 / 大循环]
3. 可维护性：检查 [命名 / 重复代码 / 过度抽象]
4. 边界情况：检查 [空值 / 错误处理 / 加载状态]

输出格式：
| 严重度 | 位置 | 问题 | 建议修复 |
|--------|------|------|----------|
```

### 技术方案设计
```
## 背景
[当前架构 / 痛点 / 业务需求]

## 要求
1. 先列出 3 种可行方案，各写一句话核心思路
2. 让我选择方向后，再展开详细设计
3. 每个方案包含：架构图(mermaid)、优缺点、风险

## 约束
- 必须兼容现有 API 接口
- 考虑 [1000 QPS / 10 万用户] 量级
```

## 三、10 条提效铁律（Anthropic 官方推荐）

| # | 规则 | 说明 |
|---|------|------|
| 1 | **给 AI 明确角色** | "你是 React 性能优化专家" > "帮我优化代码" |
| 2 | **指定输出格式** | 表格、JSON Schema、Markdown 清单 |
| 3 | **上下文前置** | 先给文件/文档，再提问题 |
| 4 | **说清楚不要什么** | "不要引入新依赖" / "不要用 any" |
| 5 | **要求分步推理** | "先思考，再回答" / think step by step |
| 6 | **给示例** | 展示"好的输出"比描述 10 句更有效 |
| 7 | **大任务拆小** | 研究 → 方案 → 评审 → 编码 → 测试 |
| 8 | **让 AI 自审** | "审视你自己的回答，找出可能的遗漏" |
| 9 | **迭代优于重来** | 在同一个对话中持续修正，不要新开会话 |
| 10 | **独立验证** | 提示词再好也不能消除幻觉，关键结果必须验证 |

## 四、10 个"魔力短语"（立即提升准确度）

```
1. "先仔细阅读相关代码，然后告诉我你的理解，等我确认后再动手"
2. "列出你的假设，逐一验证"
3. "如果你不确定，说'我不确定'，不要猜测"
4. "给出 3 种方案，各写一句话核心思路，让我选"
5. "完成前，自我审查：是否有更简单的实现？是否遗漏了边界情况？"
6. "用中文解释你的思路，代码用英文"（中英混排场景）
7. "思考可以很长，但最终回答要简洁"
8. "如果我的需求与最佳实践冲突，请告诉我"
9. "修改前先评估影响范围，列出所有会被影响的文件"
10. "只做我明确要求的事情，不要扩展范围"
```

## 五、CLAUDE.md 最佳配置（全局级）

### 推荐安装 golden-CLAUDE.md：

```bash
mkdir -p ~/.claude
curl -o ~/.claude/CLAUDE.md https://raw.githubusercontent.com/Z-M-Huang/golden-CLAUDE.md/main/CLAUDE.md
```

### 最小化模板（项目级 CLAUDE.md，~25 行）：

```markdown
## 命令
- 开发：`bun run dev`（端口 3002）
- 构建：`bun run build && bun run typecheck`
- 测试：`bun test`
- 代码检查：`bun run lint`

## 关键约定
- TypeScript strict 模式，禁止 any
- 箭头函数，camelCase 命名
- 所有 DB 访问通过 ORM，禁止裸 SQL
- 使用 `bun` 而非 npm/yarn

## 陷阱
- 本地需要 .env.local（见 docs/setup.md）
- 端口 5432 必须释放给 Postgres
- 禁止提交 .env.local

## 禁止事项
- 不要未经询问修改不相关的文件
- 不要引入新依赖
- 修改后自动运行 lint 和 test
```
